# Meshtastic Pager Mode Fork

> Неофициальная small-screen сборка Meshtastic для тех, кому нужен простой режим чтения сообщений в стиле пейджера.

Эта ветка добавляет в Meshtastic режим `Pager Mode` для маленьких экранов и старается делать это минимальными правками, чтобы в будущем было проще подтягивать изменения из upstream.

## О проекте

Этот проект был сделан заинтересованным пользователем с помощью искусственного интеллекта.

- Это не официальный релиз Meshtastic.
- Мы ни на что не претендуем и не пытаемся присвоить себе чужую работу.
- Прошивка делалась в первую очередь для себя, а потом была выложена на GitHub на случай, если она пригодится кому-то ещё.
- Если вам это полезно, берите и пользуйтесь.
- Если что-то не работает на вашем устройстве, скорее всего потребуется самостоятельная проверка, настройка или доработка под ваш сценарий.

Надеемся, что эта работа кому-нибудь пригодится.

## Что меняет этот форк

- Добавляет `Pager Mode` в `Message Action` для классического UI маленьких экранов.
- Добавляет похожий режим в `InkHUD` для поддерживаемых e-ink устройств.
- Фиксирует устройство на выбранном DM или канале, пока активен pager mode.
- Позволяет выйти из режима длинным нажатием кнопки.
- Сохраняет состояние pager mode после перезагрузки.
- Упрощает экран отображения сообщений, чтобы внимание было на тексте, а не на служебной информации.
- Улучшает читаемость длинных сообщений: увеличивает минимальный размер текста и включает вертикальную прокрутку.
- Включает поддержку русского текста для small-screen сборки этого форка.

## Для каких устройств это сделано

Этот форк рассчитан только на маленькие экраны:

- OLED-устройства Meshtastic с классическим экранным UI
- e-ink устройства на `InkHUD`

Большие TFT-интерфейсы специально не перерабатывались.

## Как работает Pager Mode

### Классический small-screen UI

1. Откройте DM или сообщения канала.
2. Откройте `Message Action`.
3. Выберите `Enable Pager Mode`.
4. Устройство останется на выбранном потоке и перестанет реагировать на обычную короткую навигацию.
5. Для выхода используйте длинное нажатие основной кнопки.

### InkHUD

1. Откройте нужный applet с DM или каналом.
2. Откройте меню и включите pager mode.
3. Текущий поток останется на экране, а автоматическое переключение на другие сообщения будет подавлено.
4. Для выхода используйте длинное нажатие.

## Поддержка русского текста

В этом форке русская локаль для маленьких экранов включена по умолчанию:

- Классический OLED UI использует `OLED_RU`.
- Варианты `InkHUD` используют локальный выбор шрифтов с кодировкой `Win1251`, когда включён русский язык.

Если позже потребуется поддерживать несколько языковых вариантов, правильнее будет вынести русский дефолт в отдельные build flavor или отдельные release-ветки.

## Коротко о замысле

Этот форк задумывался без лишнего размаха:

- сделать чтение сообщений на маленьких экранах удобнее
- не ломать остальную прошивку без необходимости
- оставить как можно больше поведения upstream без изменений
- упростить дальнейшее обновление форка

## Рекомендуемые цели сборки

Примеры:

```bash
pio run -e heltec-v3
pio run -e heltec-wireless-paper-inkhud
pio run -e t-echo-inkhud
```

Используйте тот `env`, который соответствует вашему железу.

## Как пользователю получить прошивку

Для большинства людей правильный путь это **готовые релизы**, а не сборка из исходников.

### Рекомендуемый сценарий для обычного пользователя

1. Открыть страницу `Releases` этого репозитория на GitHub.
2. Найти релиз, где указана его плата или семейство устройств.
3. Скачать готовый файл прошивки именно под своё устройство.
4. Прошить его обычным способом, как любую другую прошивку Meshtastic для этой платы.

### Какие файлы обычно выкладывать в релиз

В зависимости от устройства стоит прикладывать:

- `firmware-<env>-<version>.bin`
- `firmware-<env>-<version>.factory.bin` для первой прошивки ESP32
- другие файлы, если конкретная плата требует свой формат

### Когда действительно нужна сборка из исходников

Собирать самому нужно только если:

- под нужную плату ещё нет готового бинарника
- пользователь хочет что-то менять в форке сам
- пользователь тестирует не релиз, а свежую dev-ветку

### Практический вывод

Нет, обычному человеку не нужно обязательно ставить Visual Studio Code и повторять весь наш путь.

Если для его платы уже выложен готовый бинарник в `Releases`, ему достаточно:

- скачать
- прошить
- проверить работу

Именно поэтому для этого форка лучше выкладывать готовые прошивки под уже проверенные устройства.

## Ограничения

- Большие TFT-устройства специально не адаптировались.
- Русская локаль включена по умолчанию именно в этом форке, что удобно для этой сборки, но не обязательно подходит upstream.
- Для `InkHUD` изменения специально держались как можно ближе к исходной структуре, чтобы потом было проще обновляться.
- Это любительский community fork, поэтому часть сценариев всё ещё стоит проверять на реальных устройствах вручную.
